
## О проекте

Набор тестов для публичного API OpenLibrary (эндпоинты `/search.json` и `/isbn/{isbn}.json`).  
Цель — показать применение техник тест-дизайна к API: функциональное тестирование, классы эквивалентности, граничные значения, негативные и security-сценарии, а также проверку rate limiting.

---

## Структура репозитория

```
openlibrary-api-testing/
├── testcases.md
├── results.md

├── analysis.md
└── README.md
```

---

## Объект тестирования

Публичный API OpenLibrary:

- `GET /search.json?q=...`
- `GET /isbn/{isbn}.json`

---

## Методика тестирования

Использованы техники тест-дизайна:

- классы эквивалентности (EC)
- анализ граничных значений (BVA)
- негативное тестирование
- security-тесты (SQLi, XSS, Path Traversal)
- проверка rate limiting (50 запросов подряд)
- функциональные тесты поиска и ISBN

---

## Результаты

- Большая часть тестов — PASSED.
- 50 запросов подряд прошли успешно без rate limiting.
- Обнаружены дефекты:
  - **BUG‑01** — 500 Internal Server Error на спецсимволах в параметре `q`.
  - **BUG‑02** — 422 Unprocessable Entity при `q < 3`, поведение не задокументировано.

---

## Ключевые дефекты

### BUG‑01 — 500 на спецсимволах

Запрос:

```
GET /search.json?q=@#$%^&
```

Фактически: `500 Internal Server Error` + HTML stacktrace Python.  
Ожидание: устойчивый JSON‑ответ.

### BUG‑02 — скрытое бизнес‑правило: `q < 3` даёт 422

Запрос:

```
GET /search.json?q=a
```

Фактически: `422 Unprocessable Entity`  
Ожидание: `200 OK`, обычный поиск.

---

## Рекомендации

1. Добавить корректную валидацию символов.
2. Убрать утечку stacktrace.
3. Задокументировать минимальную длину `q`.
4. Добавить fuzzing для параметров API.
