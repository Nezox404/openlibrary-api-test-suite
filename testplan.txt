# Test Plan — OpenLibrary API

## 1. Project Name and Test Objectives
**Project Name:** OpenLibrary API Testing  
**Test Objectives:**  
- Проверка стабильности и корректности публичного API OpenLibrary (`/search.json` и `/isbn/{isbn}.json`).  
- Выявление дефектов при стандартном и нестандартном вводе (спецсимволы, короткие строки, длинные строки, Unicode).  
- Проверка безопасности API: SQL Injection, XSS, Path Traversal.  
- Проверка функциональности поиска и получения данных по ISBN.  
- Оценка производительности и устойчивости при нагрузке (rate limiting).  

## 2. Scope of Testing
**Включено в тестирование:**  
- Параметр `q` в `/search.json`: валидные запросы, невалидные, спецсимволы, Unicode, минимальная и максимальная длина, отсутствие параметра.  
- Эндпоинт `/isbn/{isbn}.json`: валидные ISBN, невалидные ISBN, слишком длинные значения.  
- Ограничение частоты запросов (rate limiting).  
- Безопасность: SQLi, XSS, Path Traversal.  

**Не включено в тестирование:**  
- Авторизация и аутентификация (API публичное).  
- Тестирование фронтенда OpenLibrary.  
- Интеграция с внешними сервисами кроме ISBN и поиска.  

## 3. Test Types
- Функциональное тестирование  
- Регрессионное тестирование  
- Интеграционное тестирование  
- Системное тестирование  
- Приёмочное тестирование  
- Security-тестирование (SQLi, XSS, Path Traversal)  
- Тестирование производительности (rate limiting)  
- Usability-тестирование не применяется  

## 4. Test Approach and Strategy
- Тестирование по тест-кейсам (`testcases.md`).  
- Ручное выполнение и автотесты (Postman collections, Python скрипты).  
- Exploratory-тестирование для нестандартных комбинаций параметров и fuzzing.  
- Фиксация результатов в багрепортах (`bugreport.md`) и аналитических документах (`analysis.md`).  

## 5. Roles and Responsibilities
- QA Engineer: планирование тестов, написание тест-кейсов, выполнение тестов, фиксация результатов  
- Backend Developer: исправление дефектов, уточнение бизнес-правил  
- Technical Writer: оформление документации API и ограничений  
- QA Lead / Test Manager: утверждение тест-плана, контроль выполнения, анализ результатов  

## 6. Test Environment
- Сервер: публичный OpenLibrary API  
- Операционные системы: Windows 10, Linux Ubuntu 22.04  
- Браузеры: не применяются (только API)  
- Инструменты: Postman, Python (requests), curl  
- Сеть: стабильное подключение к интернету  

## 7. Test Data
- Валидные поисковые строки: `"harry potter"`, `"tolkien"`  
- Невалидные строки: `"zxqwyplmno"`, `"a"`, строки с спецсимволами `@#$%^&`, длинные строки (255, 256, 5000 символов)  
- Unicode-запросы: `"гарри поттер"`  
- ISBN: валидные (`9780439554930`), невалидные (`ABCDEFG`), слишком длинные (`12345678901234567890`)  
- Наборы данных для нагрузочного тестирования: 50+ последовательных запросов  

## 8. Schedule
- Подготовка тест-кейсов: 1 день  
- Выполнение функциональных тестов: 2 дня  
- Security и нагрузочные тесты: 1 день  
- Анализ результатов и отчет: 1 день  
- Общая продолжительность: 5 рабочих дней  

## 9. Risks and Assumptions
**Риски:**  
- Возможный падение API при нестандартном вводе (спецсимволы, длинные строки)  
- Недокументированные бизнес-правила (`q < 3` → 422)  
- Возможные изменения внешнего API OpenLibrary без уведомления  

**Предположения:**  
- API доступен и отвечает на запросы  
- Документация OpenLibrary частично соответствует фактическому поведению  
- Фронтенд не тестируется  

## 10. Exit Criteria
- Все критические и высокоприоритетные дефекты выявлены и зафиксированы  
- Все тест-кейсы выполнены (PASSED/FAILED/ACCEPTABLE) и результаты оформлены  
- Нестабильные сценарии (спецсимволы, минимальная длина `q`) документированы  
- QA Lead утвердил финальный отчет о тестировании
